# RFC-090 多文档检视页面键盘导航（↑/↓ 切换文件 · Q/W 采纳/不采纳）

状态：Draft

## 背景

RFC-079 引入多文档检视页面（`MultiDocReviewView`），RFC-082 把"单文档 markdown + 锚定评论侧栏"抽成共享 `ReviewDocPane`。当前多文档页面只能用鼠标点击左侧导航切换文件、点击"采纳 / 不采纳"按钮裁决单篇。对需要快速逐篇过审（典型 fan-out 审计产物，常常几十篇）的评审者，全程鼠标点击效率低。

单文档页面（`reviews.detail.tsx`）早已有键盘语义（A/R/I 决策、`ReviewDocPane` 的 J/K 跨评论跳转）；多文档页面缺一套等价的"逐篇过审"键盘流。

## 目标

- 在多文档检视页面支持键盘逐篇过审：
  - **↑ / ↓**：切换到上一篇 / 下一篇文档（左侧导航的选中项），到首 / 尾夹紧，不循环。
  - **Q**：把当前文档标记为「采纳」。
  - **W**：把当前文档标记为「不采纳」。
- **填写检视意见时不生效**：当评论 popover 打开或正在行内编辑评论（即 `ReviewDocPane` 处于 capturing 态）、焦点在任意输入框 / 文本域 / 下拉、或决策弹窗（通过 / 迭代 / 退回）打开时，以上快捷键一律不触发。
- 快捷键可发现：采纳 / 不采纳按钮 `title` 标注按键；操作区展示一行快捷键提示。

## 非目标

- 不改动单文档页面（A/R/I、J/K 原样保留）。
- 不新增"取消裁决（回到未决）"语义——后端 `selection` PATCH 仅接受 `accepted | not_accepted`（`packages/shared/src/schemas/review.ts`），Q/W 与现有两个按钮等价（幂等 set），不引入回退到 `unselected`。
- 不改后端、不改 schema、不改数据流。
- 不引入跨文件的"全部采纳 / 全部不采纳"批量键（可作后续 RFC）。

## 用户故事

- 作为评审者，我打开一轮多文档检视，按 ↓ 逐篇浏览，对每篇按 Q 或 W 裁决，无需把手从键盘移到鼠标，直到"通过 (N/N)"可点。
- 作为评审者，当我选中正文片段、弹出评论框正在打字时，误按到 q / w / 方向键不会改变裁决或跳走文件。

## 验收标准

1. 焦点不在输入控件、无弹窗、未在填写评论时：↑ / ↓ 切换导航选中项（夹紧不循环）；切换后选中项滚动进可视区。
2. Q 使当前文档 `selection=accepted`，W 使 `selection=not_accepted`（命中既有 selection 端点，等价于点击对应按钮）。
3. 评论 popover / 行内编辑评论态、输入控件聚焦态、决策弹窗打开态：Q / W / ↑ / ↓ 全部静默。
4. ↑ / ↓ 阻止页面默认滚动；带 Ctrl / Cmd / Alt / Shift 修饰键时不拦截（不遮蔽 Cmd+W 等系统 / 浏览器快捷键、Shift+方向键文本选择）。
5. 采纳 / 不采纳按钮有按键提示 `title`；操作区有一行快捷键说明。
6. 纯函数（按键→动作映射、夹紧导航索引）+ 组件行为 + 源码锚点测试齐全。
