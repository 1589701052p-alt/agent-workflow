# RFC-164 工作组（Workgroup）——多 agent 编组协作（leader 派单 / 自由协作 · 聊天室视图）

> 状态：Draft
> 引出：用户 2026-07-10「现在想增加一个工作组界面……工作组就是把很多 agent 添加到一个工作组内，
> 然后工作组有一个 leader 可以给其他工作组成员分配工作……也可以决定是否可以让 worker 直接互相
> 能看到，从而形成 worker 之间的互相沟通，这种沟通能力可以用开关来决定」。
> 经四路并行调研（multica Squad 源码级 / CrewAI+AutoGen+LangGraph / Anthropic+Claude Code teams+
> OpenAI Agents SDK+MetaGPT+AgentScope / 本仓基建盘点）+ **五轮 20 问反问对齐**后落档。
> 完整拍板记录见文末「决策记录」。

## 背景

现有系统只有一种协作形态：**预编排工作流**——人先在画布上画好 DAG，任务按图确定性推进。
它覆盖「流程可预知」的场景（Code → Audit → Fix），但覆盖不了「流程要在运行时才能决定」的场景：
任务该拆成几块、派给谁、要不要返工，取决于中间结果，没法预先画成图。

**工作组**补上这块：把多个 agent（和人）编成一个组，组内在**运行时动态**决定谁干什么——
leader 逐轮拆解 / 派单 / 验收（leader-worker 模式），或全员对着共享任务清单自取协作（自由协作模式）。
业界同类物：multica Squad、Claude Code agent teams、CrewAI hierarchical、AutoGen teams、
LangGraph supervisor/swarm；本仓的差异化立足点仍是立项初衷——**回合制驱动一次性 CLI 进程 +
框架级确定性管道**，每个成员进程保持小而聚焦的上下文，全过程落库可观测（multica 的明显短板
恰恰是没有小队级时间线视图，本 RFC 用聊天室视图补齐）。

## 目标

1. **工作组资源**：新增第六类 ACL 资源 `workgroup`（沿 RFC-099：owner + visibility + grants）。
   组 = 名称 + 描述 + 组章程（instructions，全员每轮注入）+ 成员列表 + 模式 + 互见开关 + 轮数硬顶
   + 完成确认门开关。成员两类：**agent 成员**（引用既有 agent，各带组内 role 描述，进花名册供
   leader/同伴选人）与**人类成员**（引用平台用户）。leader-worker 模式必须指定一名 agent 成员为 leader。
2. **两种模式（v1）**，同一套成员/回合/消息/派单基建，仅「谁能派单」不同：
   - `leader_worker`：星型。leader 每轮读现状 → 产出结构化派活单 → 平台并行跑被派成员 →
     结果就绪唤醒 leader，循环至 leader 宣布完成。**leader 纯协调不动手**（协议级约束，multica
     Squad Operating Protocol 同款）；**worker 禁止转派**（拆分建议经消息/结果回 leader）。
   - `free_collab`：无 leader。共享任务清单，成员可添加任务、自取任务（self-claim）；
     **首轮全员并行唤醒**各自规划（prompt 查重约束 + 平台侧清单去重护栏）；互见恒全开；
     清单全完成且无新消息/新任务即收尾。
3. **执行模型 = 回合制**（不引入常驻进程）：leader/成员都是标准一次性进程 run；
   「派单 → 等结果 → 带上下文唤醒」复用反问机制已趟通的「暂停-回填-重跑」形态；
   leader 跨轮上下文优先 session 续接（RFC-026 `--session`/`--resume` 先例），历史重注入兜底。
4. **互见 = 三个独立开关**（组级配置，控制**每个 agent 成员每轮 prompt 注入哪些切片**；
   `free_collab` 恒全开）：
   - `share_outputs` 产出互见：注入同伴已完成派单的结果摘要；
   - `direct_messages` 定向消息：成员可 @ 同伴发消息、收到 @ 自己的消息（消息可唤醒对方）；
   - `blackboard` 公共黑板：注入组内公共消息流（预算裁剪）。
   三开关全关 = 纯星型（成员只见 leader 任务书）。
5. **聊天室 = 组任务主视图**：房间是**人看的全量视图**——任务成员（人）恒见全部消息，
   开关只影响 agent 注入。派单/结果/人话/系统事件统一为消息流；派单渲染为实时状态卡
   （排队/运行中/完成/失败，WS 推送），点卡进既有 node drawer / SessionTab 看执行现场；
   **人 @ 某 agent 成员 = 直接派单**（与 leader 派单同级，卡片标注来源=人），
   不带 @ 的消息进黑板并于 leader 下一轮注入（free_collab 则全员可见）。
6. **人类成员（v1 支持）**：可被派单（leader 派 / 人 @ 派均可）。派给人的单 = 房间待办卡 +
   收件箱提醒；交付**双形态**——在房间里回复该卡即交付（消息即结果），或点卡打开结构化交付表单。
   人类成员启动组任务时自动成为任务协作成员（获得房间访问与作答权）。
7. **人工介入面（v1 全量）**：
   - 成员反问沿用既有反问机制（`<workflow-clarify>` → 反问收件箱 → 答后回合继续）；
   - 运行中任务成员可向房间投递消息/指令（进黑板/唤醒 leader）；
   - **完成前人工确认门**（组级开关）：leader 宣布完成后任务转入待确认，人确认收尾 /
     驳回带意见再起一轮；
   - 运行中可改组配置（加/减成员、改开关等，落系统消息；中途加入不补历史——msghub 语义）。
8. **终止双保险**：自判完成（leader 宣布 / 清单收敛）+ 组级 `max_rounds` 硬顶（与 loop wrapper
   `max_iterations` 同构，触顶收尾语义在 design.md 定）。
9. **任务底座复用**：组任务**必绑 repo**（启动选 repo + 基底分支 + 目标文本），复用任务基建全套——
   per-task worktree、成员 run 各自隔离 worktree + merge-back（RFC-130）、任务生命周期/取消/恢复、
   任务成员制私有、`/tasks` 列表照常出现（带工作组徽标）。产出 = git diff + 组总结。

## 非目标（v1 明确排除）

- **轮转/接力模式**（round-robin / handoff）：mode 字段留扩展位，v1 只做上述两模式。
- **worker 转派 / 嵌套编组**（组中组、worker 再拉人）：Claude Code teams 同款约束（no nested teams）。
- **组作为工作流节点/wrapper**：本版工作组是独立入口，不嵌入 DAG 画布；后续版本再评估。
- **动态画布视图**：任务详情不做「运行时长出节点」的图视图，聊天室即主视图（图语义弱、回合语义强）。
- **repo 可选（无 worktree 组任务）**：纯调研组也走绑 repo 路径（不写文件即可），不做无 worktree 任务变体。
- **组级 token/成本预算顶**：终止仅轮数硬顶；token 统计基建另立项。
- **广播 @all 原语、消息已读回执、房间内富文本编辑器**等聊天产品级功能：v1 消息为 markdown 文本 + @提及。
- **不动既有工作流/任务/反问/评审的任何现存语义**：工作组是增量子系统，现有路径零回归。

## 用户故事

1. **建组**：我在 `/workgroups` 新建「支付模块攻坚组」，加入 agent 成员 `coder-a`（role：后端实现）、
   `coder-b`（role：前端实现）、`auditor`（role：代码审计），加入人类成员「我自己」（role：产品把关），
   指定 `planner` 为 leader，模式 leader-worker，打开「产出互见」，关掉「定向消息」「公共黑板」，
   max_rounds=20，打开完成确认门。
2. **启动**：我点「启动任务」，选 repo `payment-service` + 基底分支 `main`，目标写
   「把支付回调改成幂等，补齐测试」。任务创建，聊天室出现系统消息（组配置快照）与首轮 leader 运行卡。
3. **观测**：leader 第一轮产出三张派单卡（@coder-a 改回调 / @auditor 摸现状 / @我 确认幂等键口径）。
   两张 agent 卡实时显示运行中；我的待办卡出现在房间和收件箱。我在房间里回复我的卡「用订单号+事件类型」，
   即交付。coder-a 完成后结果摘要落进房间，我点卡看它的完整执行现场。
4. **介入**：中途我发现方向偏了，直接在房间发消息「回调表已废弃，改走 events 表」（不带 @）——
   leader 下一轮注入该指令修正派单；我也可以直接 `@coder-b 顺便把前端超时提示改了`——生成一张
   来源=人的派单卡，无需经过 leader。
5. **收尾**：leader 判定全部完成，宣布收尾；因确认门开启，任务转待确认。我看 diff 和组总结后点确认，
   任务 done；若驳回并写「测试没覆盖并发场景」，leader 带意见再起一轮。
6. **自由协作**：另一个组选 free_collab 模式跑「全仓 TODO 清理」：首轮全员并行唤醒、各自往清单
   添加任务并认领，互见全开，清单清空且无新消息后任务自动收尾。

## 验收标准

- AC-1 资源面：workgroup CRUD + ACL（未授权 404/列表过滤，与五类资源同形）；成员/leader/开关/
  轮数/确认门可配；表单校验（leader_worker 必有 leader；free_collab 开关恒开且不可关）。
- AC-2 leader-worker 回合闭环：leader 派活单 → 成员并行 run → 结果回流 → leader 唤醒续派/宣布完成；
  全链路落库（回合、派单、消息可追溯）；leader 纯协调（无代码写入产出）。
- AC-3 free_collab 闭环：首轮全员唤醒、添加/认领任务去重、事件驱动续轮、清单收敛自动收尾。
- AC-4 互见开关：三开关逐一独立生效（对 agent prompt 注入的影响可测——纯函数断言注入切片）；
  全关=纯星型；free_collab 恒全开。
- AC-5 聊天室：消息流全量呈现（人恒见）；派单卡实时状态（WS）；点卡直达 run 详情；
  人 @ 成员=直派、不 @ =黑板/leader；房间输入框可用。
- AC-6 人类成员：派单生成待办卡+收件箱条目；房间回复即交付、表单交付两形态皆可；
  交付结果进入下一回合上下文。
- AC-7 反问沿用：组内成员 run 发 `<workflow-clarify>` 走既有反问收件箱与回合恢复，无分叉路径。
- AC-8 终止：leader 宣布完成收尾；max_rounds 触顶按 design.md 语义收尾；确认门开启时先进待确认，
  确认/驳回行为符合目标 7；确认门泊住期间**不触发**既有卡住检测/自动修复误报（生命周期不变式
  兼容，design §8.2）。
- AC-9 中途改配置：加/减成员、改开关生效于后续回合并落系统消息；中途加入成员不补注入历史。
- AC-10 复用与隔离：组任务复用 worktree/merge-back/生命周期；现有工作流任务路径零回归
  （全量门禁 + 既有测试全绿）。
- AC-11 门禁：`typecheck + lint(0 warn) + test + format:check` 全绿 + binary smoke + 前端 vitest；
  Codex 设计门与实现门；新增 UI 全部复用公共组件/样式（Dialog/Form/Select/StatusChip/.btn 等），
  视觉对齐自查。
