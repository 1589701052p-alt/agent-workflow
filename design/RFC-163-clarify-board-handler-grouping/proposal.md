# RFC-163 反问看板改派卡分组呈现（下发前一问一卡 · 下发后各处理节点拆开）

> 状态：Draft
> 引出：用户 2026-07-10「为什么在修改处理节点的时候，会创建一个新的节点而不是直接显示修改掉了」→
> 解释 RFC-162 的「提问节点必然重跑、改派=增一个上游处理节点」模型后，用户拍板：
> **「只有下发后才会变成 2 张卡，下发前都是一张卡」**。

## 背景

RFC-162 归一后，一条 clarify 问题默认挂**一个处理节点**（提问节点自己：self / questioner）。
「让上游改」= 人工改派 → **增一条 `designer` 处理条目**、并保留提问节点条目（提问节点必然重跑
才能续出产出，删不得，否则 strand）。因此改派后该问题有**两个处理条目**。

当前看板 `TaskQuestionList` 是**一条目一张卡**（`col.map(entry)`）：改派后立刻出现**两张卡**
（提问卡 + 上游卡），视觉上像「凭空多了个节点」——这正是用户的困惑点。模型本身正确（两条目
＝两次必然重跑），但**呈现**不直观。

## 目标

- **下发前**（待指派 / 待下发）：同一问题的多个**未下发**处理条目**收拢成一张卡**——主体是问题
  本身，卡内列出它的处理节点（提问节点自己 + 你增派的上游/下游）。改派＝在**同一张卡内增/删一行
  处理节点**，不再像「多冒出一张卡」。
- **下发后**（处理中 / 已处理待确认 / 完成）：每个已下发处理条目**各自独立成卡**——因为它们此时
  在图里各自独立重跑、需要分别追踪与确认（提问卡与上游卡的重跑时机、产出、确认都是独立的）。
- **顺带修正一个 board 路径隐患**：把「下发前整组一起 stage / 一起批量下发」作为卡级动作，确保
  提问节点与上游处理节点**一起走统一 frontier 下发**（board 手动下发路径此前无 RFC-162 Finding-2
  的「提问卡脱离上游单独下发」保护——只 stage 提问卡再下发会让它脱离上游先跑、乱序）。

## 非目标

- **不改后端 / 数据模型 / dispatch 语义**（**纯前端**）：条目仍是 `task_questions` 行，
  `reassign`/`stage`/`dispatch`/`confirm` 端点契约全部不变；分组纯前端呈现 + 卡级动作编排（stage
  循环调既有 per-id 端点）。Codex 设计门 P2（「改派仅下发前」非硬不变式）实现期勘误为**降级不变式**
  ——「已下发 asker + 未下发 designer」＝答后修订流常态（新 designer 自成待指派单卡、下发经级联再跑
  asker），初版守卫因禁掉该一等流程打红 19 测而撤销；详见 design §3.5。
- **不改 `/clarify` 详情页**、集中作答面板、画布、决策流。
- **不引入跨列合并**：分组只发生在「同为未下发」时（见 design——改派仅限下发前 + 整组一起下发 ⇒
  不会出现「一半下发一半没下发」的分裂，故无需跨列拼卡）。
- 不改手工问题（单 designer、无提问节点共存）——它天然是「一个的组」＝仍单卡。

## 用户故事

1. 我在待指派看到一条问题一张卡。点「改派…」选一个上游 agent → **同一张卡里多出一行「上游 X（修订）」**，
   卡还是这一张（不是冒出第二张）。
2. 我点这张卡的「加入待下发」→ 卡整体进待下发（它的提问节点 + 上游行一起进）。
3. 我点「批量下发」→ 这一组一起走统一 frontier 下发；**下发后**看板里它按处理节点**拆成各自的卡**
   （处理中/已处理待确认里，提问节点一张、上游一张），我分别看它们的进度、各自确认。
4. 若我把处理节点改回提问节点自己 → 卡里那行「上游」消失、回到一行。

## 验收标准

- AC-1 下发前：同一问题的未下发条目恒收拢成一张卡；改派增/删的是**卡内一行**，卡数不变。
- AC-2 下发后：已下发条目各自单卡（与今日一致），可分别确认。
- AC-3 stage / 批量下发为**组级**：一张未下发卡的「加入待下发」使其全部处理节点一起 staged；批量下发
  把该组全部处理条目一起下发（统一 frontier）。**不可**只下发提问节点而遗漏共存上游。
- AC-4 手工问题、单处理节点问题：仍单卡（组内一员），行为不变。
- AC-5 节点 filter / 计数：按处理节点计数不变；filter 命中任一处理节点则展示整组卡。
- AC-6 门禁四项 + 前端 vitest；Codex 设计门 + 实现门。
- AC-7 视觉自查：分组卡复用既有 `Card` / `.btn` / `StatusChip` 等公共原语，与其余看板风格一致。
