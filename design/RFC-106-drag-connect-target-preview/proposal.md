# RFC-106 — 拖拽连线的目标预览与「精准复用输入」反馈

状态：Draft

## 背景

工作流编辑器（xyflow v12）里，从一个节点的**输出端口**拖一根连线去接另一个节点时，当前交互有个误导点：

- 节点左侧入口由 `PortHandles` 渲染成两层 —— 一个铺满左缘的**隐形 catch-all**（`__inbound__`，落点 = 新输入端口）+ 每个已有输入端口一个**具名 handle**（落点 = 复用该端口）。
- 具名 handle 命中优先级高（z-index 1 > catch-all 0），且 xyflow 默认 `connectionRadius` 20px 的磁吸较"贪"。
- 结果：当目标节点**已经有一个输入**时，拖拽过程中的临时线会**先吸附到那个已有输入端口上**，用户以为在复用它；而真正的落点端口要**松开鼠标后**才显示出来。

这个"拖拽时所见 ≠ 松手后所得"导致一类高频误操作：**两条线被不经意地连到了同一个输入端口**（同一 `{{port}}` 被两个来源占用）。用户最初把它描述为"两条线不能直接拖到同一个输入节点"，但澄清后确认：**不需要硬性禁止 fan-in，只要给出直观的实时反馈，让人一眼看清落点，就能避免误连**（用户原话："渲染可以解决这种错误连线问题，只要让使用人有直观反馈就行了"）。

## 目标

针对手势 **"从输出端口拖出 → 落到一个已存在的节点上"**（用户确认的范围）：

1. **拖拽过程中实时渲染"将要连上的目标"**：高亮当前悬停的目标节点，连线吸附到真实落点，所见即所得（不再是松手才显示）。
2. **默认落到"新输入"**：大致拖到节点入口区（未精准命中已有端口）时，落点是一个**新的输入端口**，并提示"新输入"。
3. **复用已有输入需精准**：要复用某个已有输入端口，必须**精准拖动到该端口上**（更小的磁吸命中区）；命中时**高亮该端口并明确提示"将复用同一输入"**。
4. 反馈的视觉状态与最终落点**始终一致**（消除"拖时吸附在 A、松手却落在 B"的偏差）。

## 非目标

- **不做硬性校验阻断**：不新增"禁止两条边进同一端口"的 validator 报错；本 RFC 是纯交互/渲染增强，靠反馈而非阻断。
- **不改运行时调度 / 数据语义**：`buildScopeUpstreams`、prompt 注入、fan-in 解析等一律不动。
- **不改 clarify / cross-clarify channel 拖拽**：那套由 `classifyClarifyConnection` / `classifyCrossClarifyConnection` 专门处理，且刚由 RFC（false-root 修复）收紧；本特性在它们之后生效，不触碰 channel 系统口。
- **不改 output 节点"收集多路"的数据模型**：output 仍可收集多个不同来源；本特性只让它新增端口 / 复用端口的**默认行为与反馈**和其它节点一致（范围含 output，见下）。
- 不改"拖到空白画布新建节点"等其它手势（本次只覆盖"落到已存在节点"）。

## 适用范围

用户意图是"所有目标节点（含 output）"。落到具体渲染器后按"误连是否会发生"分两层（Codex 设计 gate P2 校准）：

- **"新输入默认 + 精准复用 + 唯一端口名"核心机制** —— 适用于用 `PortHandles` catch-all 接收**任意具名输入**的节点，即 **agent-single 与 output**。这正是"两条线误连同一输入"真实发生的地方。
- **实时目标反馈（高亮 + 连线预览）** —— 对**所有** `toNode` 通用（数据源是 xyflow `useConnection`），review / wrapper-fanout 拖上去也会高亮、给预览。
- **专用输入模型不套核心机制，零回归**：
  - `wrapper-loop` / `wrapper-git`：v1 校验器明令"不接受入边"（`edge-target-port-missing`），根本不是可落点 → N/A。
  - `review`：单一 `__review_input__` 绑定输入（RFC-007，经 inputSource），不存在"两个任意输入撞名" → `classifyDropTarget` 返回 `none`，沿用既有 review 接线。
  - `wrapper-fanout`：内联 boundary handle（`WrapperNodes.tsx`，非 PortHandles）+ `inputs[]` 模型 → `classifyDropTarget` 返回 `none`，沿用既有 boundary 接线；其多源是否撞名由 fanout 自己的 `inputs[]` 规则管，列为后续可选增强。
- clarify/cross 的系统 handle 一律不在此列（被前置 classifier 拦走）。

## 用户故事

1. 作为编排者，我从 `A.out` 拖线，把鼠标移到已有一个输入的节点 `C` 的入口区（不精准压在已有端口上）：`C` 高亮、连线预览指向 `C`、并显示徽标"新输入"；松手 → `C` 多出一个新输入端口接住 `A.out`，原有输入不受影响。
2. 作为编排者，我确实想让 `A.out` 复用 `C` 已有的输入端口 `x`：我把连线**精准拖到 `x` 那个小端口点上**，它被高亮且提示"复用同一输入"；松手 → `A.out` 接到 `C.x`（替换/复用该端口的来源）。
3. 作为编排者，我大致拖到 `C` 上但没压准 `x`：默认走"新输入"，**不会**被误吸到 `x` 上 —— 误连消失。

## 验收标准

- [ ] 拖拽中（未松手）即可看到：目标节点高亮 + 连线指向它 + "新输入 / 复用同一输入"二选一的明确提示。
- [ ] 落到已有输入端口需要"精准命中"（其磁吸半径明显小于节点入口区的默认 catch-all）；非精准悬停一律判为"新输入"。
- [ ] 预览提示的"新输入 / 复用"判定，与松手后实际产生的边**完全一致**（同一判定函数驱动两者）。
- [ ] 复用落点 → 边接到该已有端口；新输入落点 → 产生一个新的、与已有端口**不冲突**的输入端口名。
- [ ] clarify/cross channel 拖拽、wrapper-fanout 边界边、output 多路收集等既有行为**零回归**。
- [ ] 判定逻辑抽为纯函数并单测；渲染层保留至少一条 role/文本断言 + 源码锚点断言（catch-all 优先级、复用小磁吸 class）。
- [ ] `bun run typecheck && 前端 vitest && format:check` 全绿；CI 全绿；Codex 设计 gate + 实现 gate 通过。

## 开放问题（实现期定）

- "精准复用"的磁吸半径具体取值，需起 dev server 手动拖拽微调到"手感正确"（plan 列为显式验证步骤）。
- "复用同一输入"提示的承载形式（连线旁徽标 vs 端口行高亮 vs 两者）—— 倾向复用既有 canvas 样式原语，design 给初版，截图自查后定。
