# RFC-085 — 方法调用链 / 时序图（Method Call Chain & Sequence Diagram）

状态：**Draft**（需求已四轮澄清定稿，待用户批准进入实现）

> 编号说明：用户口头说"起草 RFC-084",但 `RFC-084-conformance-auditor` 已被并行协作者占用,本 RFC 顺延为 **RFC-085**。

## 背景

RFC-083 给出了结构化 diff 的**类协作图**(类/方法卡片 + 继承/引用/调用边 + 方法级高亮),回答"谁和谁有关系"——一张无向的关系网。

但 review 一个 agent 改动时,常要问另一类问题:**"这个方法点下去,后续会一路调到哪些方法?"**——一条**有方向、可递归穿透、能读成执行骨架**的**调用链**;最自然的呈现是**时序图**(参与类作 lifeline,调用作自上而下的有序消息)。

RFC-083 的数据**不足以**直接做:`impact` 是**反向**的(「谁调了我」)且稀疏;按名匹配只产出无序对。本 RFC 增加**正向、可穿透到未改动代码、按需懒加载**的调用解析 + 调用链遍历 + 时序图。

## 目标

1. 在结构视图**点一个改动方法**(树视图成员行 / 关系图方法行的小入口),进入**第 5 个标签「调用链」**,以它为根看**后续正向调用链**。
2. 调用链**穿透全仓**(经过未改动代码),呈现真正的执行骨架;**按需懒加载**——展开一层才解析那层涉及的文件,结果缓存。
3. 默认展开 **1 层**(根 + 直接被调),其余 `▸` 手动逐层展开;带**环检测**与**深度/节点上限**。
4. 算"调用" = **方法调用 + 构造函数(`new X()`)**(构造也跑代码、是执行链一环)。
5. 提供**时序图视图**(阶段 3):lifeline + 自上而下按源码顺序的消息 + 激活。
6. **静态、确定性、可复现优先**(同 RFC-083);**尽力而为 + 断点明标**:接口/虚函数/反射/链式等静态解不开的,标"未解析"灰显、不下钻、**绝不臆造**。
7. **全 8 种语言**(C++/Java/Python/Rust/Go/JS/TS/Scala):抽调用全覆盖;**解析精度按语言分档**(静态类型链较全,Python/JS/C++/Scala 多断点),深度模式 SCIP 可用时任意语言精确。
8. **复用 RFC-083**:符号集、成员范围、深度 SCIP、注释/字符串剥离——不另起提取管线。

## 非目标(本 RFC)

- **不做运行时/动态追踪**:纯静态,不插桩、不跑代码。
- **不追求调用图完整性**:完整度受语言静态性 + 提取器(基线/SCIP)限制,显式标注、不是 bug。
- **不预先全仓建索引**:懒加载,只为你看的那条链付费(轻量"类名→文件"索引除外,见 design)。
- **入口只限改动方法**:v1 不做"任意方法搜索/选择器"当根(穿透到未改动是链的下游,不是入口)。
- **不替代 RFC-083 关系图**:时序图是从树/图进入的**第二视图**。
- **不引入 LLM 摘要**:后置可选增强。

## 用户故事

1. **追主流程**:点 `GameEngine.tick()` 旁的「调用链」图标 → 第 5 标签以 tick 为根,显示它直接调的 `move()` / `render()`(各带 `▸`);点 `▸` 展开 `move()`,即便 move 未改动也懒解析它的下一层。
2. **穿透未改动**:链不止于本次 diff——`charge()`[改] → `repo.save()`[未改] → `db.exec()`[未改],一路下钻看完整执行骨架。
3. **断点可信**:`handler.run()`(接口)、`factory.get().x()`(链式)解不开 → 灰显"未解析"、不下钻;能解析的精确连。
4. **构造也跟**:`new Receipt()` 作为一个链节点,展开看构造函数里又调了谁。
5. **时序图**(阶段 3):同一条链切到时序图,参与类作 lifeline、消息按源码顺序自上而下,当"执行剧本"读。

## 验收标准(产品层)

- 点一个改动方法 → 第 5 标签以它为根,列出**直接被调**(方法 + 构造)的、能静态解析到具体方法的调用(默认 1 层)。
- `▸` 懒展开下一层(含穿透到**未改动**文件);递归 ≥3 层;**环检测**(A→B→A 不死循环);**深度/节点上限**命中显式标"已截断"。
- 解析三档:`resolved`(定位到具体方法)/ `external`(到类未到方法)/ `unresolved`(灰显);**未解析绝不臆造目标**。
- 8 种语言都能出链;静态类型(Java/Go/TS)链较全,动态(Py/JS)+ C++/Scala 多断点——**断点显式标注**;深度 SCIP 可用时更准、不可用自动回退基线。
- 时序图(阶段 3):消息按源码顺序;lifeline 去重;至少覆盖 Java + TS/JS。
- 全绿门槛:`typecheck && test && format:check`(+ 单二进制 smoke,动 shared/后端)。

## 决策记录(用户四轮澄清,2026-06-06）

| 维度 | 决策 |
| --- | --- |
| 调用链范围 | **全仓穿透**(经过未改动代码,真正执行骨架) |
| 提取方式 | **按需懒加载**(展开一层才解析,缓存);不预先全仓建图 |
| 入口(根方法) | 只**本次改动的方法**;从树/关系图方法行的**小入口图标**进入(不覆盖原有点行=跳 hunk/高亮) |
| 视图落点 | 结构页**第 5 个标签「调用链」**,以点中方法为根 |
| 默认展开 | **1 层**(根 + 直接被调),其余 `▸` 手动 |
| 算"调用" | **方法调用 + 构造函数(`new`)** |
| 解析缺口 | **尽力而为 + 断点明标**(接口/虚/反射/链式 → unresolved 灰显,不臆造) |
| 呈现形态 | 先**调用树**(阶段 1-2),后**时序图**(阶段 3) |
| 时序图渲染 | mermaid vs 自绘 SVG —— **留 PR-C 拍板** |
| 语言覆盖 | **全 8 种,尽力而为**(静态类型链较全;Py/JS/C++/Scala 多断点;深度 SCIP 补强) |
| 跨文件解析 | 轻量"类名→文件"索引 + 按需 parse(技术细节,实现者定);深度模式用 SCIP |
