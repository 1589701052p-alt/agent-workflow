# RFC-168 工作组详情页 UX 重构——成员为主体 + 上下文面板

状态:Draft(待用户批准)
日期:2026-07-11
前置:RFC-164(工作组)、RFC-166(能力卡)、RFC-167(dynamic_workflow 第三 mode)均已 Done。

## 1. 背景

用户反馈(2026-07-11 原话):

> 工作组的界面太丑了,编辑agent应该是这个界面的主要能力,而现在被放在了一个边缘位置,你需要重新设计ux,让操作更简洁,符合操作思路,界面要直观

结合追问澄清,「编辑 agent」指**编辑工作组成员**(别名 / 角色描述 / leader / 增删),
不是编辑 agent 全局定义(用户原话:「我说的编辑agent是编辑成员」)。

### 现状问题(`/workgroups/$name`,RFC-164 PR-1 形态)

页面自上而下:header 操作行 → readiness banner → 「基本信息」(描述 + 章程大
textarea)→「协作模式」→「协作开关」(3 switch + 最大轮数 + 完成门)→ **成员卡片区
(页面最底部)**。

1. **主次颠倒**:配置(描述/章程/开关——建组后低频修改)占满整个主视区;成员——工作组
   的灵魂、最高频的操作对象——被压在最底部,首屏根本看不见。
2. **全宽表单丑**:单列文档流里每个输入框都拉满 ~1500px 宽,视觉空旷失衡。
3. **成员卡信息贫瘠**:只有别名 + 类型 chip + agent 名 + 角色描述一行;RFC-166 引入的
   agent 能力卡只在「添加成员」对话框里预览一次,添加完就再也看不到。
4. **成员编辑要过弹窗**:改个别名/角色也要 Dialog 一开一关;添加成员同样。操作路径与
   「配团队 → 逐个调成员 → 定协作规则 → 启动」的心智不匹配。

## 2. 用户确认的决策(2026-07-11 三问)

| # | 决策 | 内容 |
| --- | --- | --- |
| D1 | 布局 = **上下文面板式** | 主区为成员卡片画廊;右侧常驻上下文面板——未选中成员时显示**组配置**,点选成员时切换为**该成员的编辑器**。与工作流编辑器(画布 + NodeInspector)心智一致。 |
| D2 | 编辑深度 = **成员属性 + 只读能力卡** | 面板内编辑的是成员属性(别名/角色描述/设 leader/移除);agent 定义本身只展示只读能力卡 + 「编辑 agent 定义」跳转链接(`/agents/$name`)。 |
| D3 | (由 D2 推导)| 不引入 agent 定义修改入口,无共享资源分叉语义问题。 |

## 3. 目标

1. **成员为主体**:进入页面第一眼看到完整团队阵容——谁在队里、什么角色、什么能力、
   谁是 leader。
2. **成员编辑零弹窗**:点卡片 → 右侧面板内联编辑;添加成员也走同一面板位置;整页
   仅保留删除组 / 重命名两个既有 Dialog。
3. **agent 能力常驻可见**:agent 成员卡带端口摘要 chips;面板内展示完整能力卡
   (复用 `AgentCapabilityCard`)。
4. **视觉修缮**:两栏工作台布局,消除全宽字段与失衡留白;卡片升级;明暗双主题对齐。
5. **保存语义不变**:组配置 = draft + header 保存;成员操作 = immediate PUT
   (read-current → pure change → PUT 全量替换,沿用 `lib/workgroup-form.ts` 纯函数)。
6. **保存后留在原地**:组配置保存成功后不再跳回列表页(现状会 `navigate('/workgroups')`,
   在「工作台」形态下不合理),改为按钮态「已保存」反馈(沿用现有 i18n `saved` 文案先例)。

## 4. 非目标

- 不改后端 / API / schema——纯前端 RFC,零 migration。
- 不改 agent 定义编辑能力(仍在 `/agents/$name`)。
- 不改工作组列表页、QuickCreate 创建流、任务房间(`WorkgroupRoom`)、任务详情的
  动态工作流面板(`DynamicWorkflowPanel`)。
- 不改 mid-run 配置对话框(`WorkgroupTaskConfigDialog`)的交互——它复用的
  `AgentMemberDialog` / `HumanMemberDialog` Dialog 壳保留。
- 不改三种模式的行为语义(leader_worker / free_collab / dynamic_workflow 的字段
  显隐与校验逻辑照旧,只换容器)。
- 不做成员拖拽排序、批量操作(后续有需要另立 RFC)。

## 5. 用户故事

1. 我打开工作组详情页,首屏即看到全部成员卡片:别名、leader 徽章、agent 名、角色
   描述、输入/输出端口摘要;右侧面板显示组配置。
2. 我点选成员「coder」,右侧面板切换为:别名、角色描述输入框,「设为 leader」
  (leader_worker 模式且非 leader 的 agent 成员)、「移除成员」,下方是 coder 的完整
   能力卡与「编辑 agent 定义 →」链接;我改完别名点「保存成员」,立即 PUT 生效。
3. 我点「+ 添加 agent 成员」,右侧面板变为添加表单:agent 选择(带实时能力卡预览)、
   别名(自动跟随 agent 名)、角色描述;确认后成员入列并保持选中该新成员。
4. 我没选中任何成员时改协作开关、章程,点 header「保存」,按钮短暂显示「已保存」,
   页面留在原地。
5. 我把窗口缩窄(<960px),页面降级为单列:成员画廊在上,上下文面板内容在下,
   功能不缺失。

## 6. 验收标准

- [ ] 详情页为两栏工作台:左成员画廊(主区)+ 右上下文面板(sticky);<960px 降级单列。
- [ ] agent 成员卡显示:别名、leader 徽章(仅 leader_worker)、agent 名、角色描述、
      输入/输出端口 chips;引用了已删除 agent 的成员卡显示悬空警示。
- [ ] 点选卡片 → 面板切换为成员编辑器;再点同卡 / 面板关闭 / Esc(面板内,不与
      Dialog 抢层级)→ 回组配置;激活/关闭/移除/添加的键盘焦点迁移完整(design §6)。
- [ ] 成员编辑器含:别名、角色描述、保存成员(immediate PUT,single-flight)、
      设为 leader、移除(二次确认),agent 成员另有只读能力卡 + `/agents/$name`
      跳转链接;失败草稿保留可重试,错误按面板归属。
- [ ] 添加 agent / human 成员走面板(不弹 Dialog);dynamic_workflow 模式无 human
      入口(现状保留);确认后新成员保持选中(wire 规范化复合键匹配)。
- [ ] 组配置保持 draft + header 保存;保存成功后**留在详情页**并给「已保存」反馈
      ——保存期间继续编辑时不误标「已保存」;draft 模式与成员不兼容(如 dyn+human)
      时错误就地可见而非只禁用按钮。
- [ ] `WorkgroupTaskConfigDialog`(mid-run)行为完全不变:既有测试断言全保持,并按
      design §8.1 增量锁定 Dialog 壳行为契约(Human 全链/嵌套关闭等)。
- [ ] `lib/workgroup-form.ts` 纯函数变换零改动(状态机层不动,纯视图重构;仅允许
      mode 兼容错误 key 增补)。
- [ ] `bun run typecheck && bun run lint && bun run test && bun run format:check` 全绿;
      明暗双主题视觉自查通过。
